## Questions de vocabulaire et de culture informatique

### Question 5 – Ajustement de la pré-configuration

#### 1. Quelle est la différence entre `d-i pkgsel/include` et `tasksel tasksel/first` ?

- **`d-i pkgsel/include`**  
  → Directive **preseed** utilisée pendant la phase d'installation des paquets de base (phase `pkgsel`).  
  → Permet d'installer des **paquets individuels** de manière très précise.  
  → Syntaxe : on liste simplement les noms des paquets séparés par des espaces.  
  → Exemple :
    ```
    d-i pkgsel/include string sudo git curl sqlite3 bash-completion fastfetch
    ```
  → Utile quand on veut des paquets très spécifiques qui ne font pas partie d'une tâche prédéfinie.

- **`tasksel tasksel/first`**  
  → Directive qui indique à `tasksel` quelle(s) **tâche(s)** installer.  
  → Les tâches sont des **ensembles prédéfinis** de paquets (ex. : environnement de bureau, serveur web, etc.).  
  → Syntaxe : on utilise `multiselect` pour en choisir plusieurs.  
  → Exemple :
    ```
    tasksel tasksel/first multiselect mate-desktop
    ```
  → Utile pour installer rapidement un environnement complet (MATE, GNOME, XFCE, serveur SSH, etc.).

**Résumé en tableau** :

| Critère                     | `d-i pkgsel/include`                  | `tasksel tasksel/first`              |
|-----------------------------|----------------------------------------|---------------------------------------|
| Niveau de granularité      | Très fin (paquets unitaires)          | Plus global (groupes de paquets)     |
| Type de contenu             | Paquets nommés individuellement       | Tâches prédéfinies (desktop, etc.)   |
| Exemple d'utilisation       | sudo, git, curl, fastfetch            | mate-desktop, xfce-desktop           |
| Quand l'utiliser ?          | Besoin précis, paquets hors-tâche     | Installation d’un environnement      |

#### 2. Le preseed contient-il des informations sensibles (mots de passe) ? Comment devrait-on protéger ce fichier en production ?

**Réponse : Oui**

Le fichier preseed contient généralement :

- Le mot de passe **root** (souvent haché avec `md5`, `sha512`, ou en clair dans certains cas)
- Le mot de passe de l’utilisateur standard (idem)
- Parfois des clés SSH, des noms d’utilisateur, des configurations proxy, etc.

Exemple typique dans un preseed :

```text
d-i passwd/root-password-crypted password $6$...
```

**En production, comment protéger ce fichier ?**

| Mesure de protection                  | Niveau | Commentaire                                                                 |
|---------------------------------------|--------|-----------------------------------------------------------------------------|
| Permissions strictes                  | ★★★    | `chmod 600 preseed.cfg` – seul le propriétaire peut le lire                |
| Propriétaire root                     | ★★★    | `chown root:root preseed.cfg`                                              |
| Stockage chiffré                      | ★★★★   | Stocker sur un disque chiffré (LUKS) ou dans un vault (HashiCorp Vault)   |
| Chiffrement du fichier lui-même       | ★★★★   | Chiffrer avec GPG ou age avant diffusion                                   |
| Ne jamais versionner en clair         | ★★★★★  | Ne pas commiter le preseed dans Git sans chiffrement (utiliser git-crypt) |
| Utiliser des secrets externes         | ★★★★★  | Remplacer les mots de passe par des variables injectées (Ansible, Packer)  |
| Accès restreint (serveur http/https)  | ★★★★   | Servir via un serveur avec authentification + HTTPS                        |

**Meilleure pratique recommandée en 2025-2026 :**  
Ne plus mettre de mots de passe en clair ou hachés dans le preseed → utiliser un mécanisme d’injection dynamique (ex. : cloud-init, Ansible, Terraform avec secrets manager).
